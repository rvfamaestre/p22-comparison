\section{Current Progress, Results and Perspectives}

The project is now presented around a single main implementation, \texttt{mustykey}. The recent work kept only the elements that are directly useful for the current workflow on the DCE cluster: GPU-assisted training for faster experiments, checkpoint restoration, and a cleaner evaluation export. The parallel \texttt{rafa} branch is not part of the main report anymore.

\subsection{Current status}

The latest campaign was run on the CentraleSup\'elec DCE infrastructure with an NVIDIA RTX~2080~Ti. Training used the vectorized backend, while evaluation still used the standard simulation path so that testing remained aligned with the controller used at inference time.

The campaign contains two training runs at 50\,\% human drivers, one with PPO and one with SAC, plus nine evaluation runs: No~RL, PPO, and SAC, each tested at 25\,\%, 50\,\%, and 75\,\% human-driver ratios. With 8 parallel environments, 25 training batches correspond to 200 physical episodes.

\subsection{Evaluation results}

For the report, we do not rely on a single ``top metric''. Mean speed over the last 100 steps is useful because it focuses on the stabilized regime, but it should be read together with late-run variance and safety indicators. Otherwise, a small speed gain can hide the fact that the traffic remains unsafe or too oscillatory.

\subsection{Interpretation}

Three conclusions are enough to summarize the current state. First, 25\,\% human drivers is already an easy regime for the baseline controller, so RL has little room to improve the result. SAC slightly lowers the late-run variance, but not enough to justify a strong claim. Second, 50\,\% human drivers is the only regime where learning produces a clear benefit: SAC gives the best compromise, with a lower stabilized variance and essentially unchanged stabilized mean speed. Third, 75\,\% human drivers remains unresolved. PPO is slightly less bad than the other methods, but every controller still triggers collision clamps.

The most important limitation is therefore not training speed anymore but result credibility. These experiments come from a single evaluation seed, so they are useful for matched-condition comparison, not for a general scientific claim. In practice, the next step should not be ``train longer'' by default. It should be ``evaluate the saved checkpoints on several unseen seeds and then decide whether retraining is needed''.

\subsection{Future perspectives}

\begin{itemize}
    \item run a multi-seed evaluation of the saved PPO and SAC checkpoints before launching any new long training campaign;
    \item tighten the early-stopping patience so that it matches a 25-batch training run instead of acting only as passive logging;
    \item strengthen the safety treatment during learning, because the present soft-penalty design is not sufficient in the 75\,\% human-driver regime;
    \item revisit the reporting logic so that no single objective metric is presented as decisive without the corresponding safety context.
\end{itemize}

At this stage, the project already supports one clear statement: the residual RL controller is promising in the intermediate mixed-traffic regime, but the difficult high-HDV case still requires a safer formulation and a stronger validation protocol.
